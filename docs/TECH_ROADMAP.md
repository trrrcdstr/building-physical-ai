## 二、Phase 1：数据引擎（当前→6月）

**核心命题**：证明工程级真值数据能让机器人在建成环境安全干活。

**报告对齐**："数据并非越多越好，真正稀缺的是'可用数据'——可检索、可对齐、可跨本体复用。"

### 2.1 数据治理攻坚

| 模块 | 当前状态 | Phase 1目标 | 技术路径 | 关键指标 |
|------|----------|-------------|----------|----------|
| CAD全量解析 | 5个房间测试，76个DWG待转换 | 完成76个DWG全部解析，覆盖3个地产项目 | ①ODA工具链解决DWG→DXF ②建立图层映射规则库 ③图纸质量自动检测 | 解析成功率>90%，墙体/门窗识别率>95% |
| 四层空间模型 | Block/Object层成熟，Road层基础版，Harness层60% | 五层全部产品化 | ①A*路径规划补全Road层 ②机电图解析引擎 ③规范规则引擎 | 路径规划成功率>95%，约束检查准确率100% |
| 材料物理属性库 | 22种材料 | 扩展至100+种，覆盖常用建材+家具 | 材料商数据录入+规范文档解析+人工校验 | 属性准确率100% |
| 虚实对齐 | 手动对齐，误差~10cm | 半自动对齐，误差<2cm | ①ICP特征点配准 ②二维码辅助定位 ③偏差可视化 | 对齐误差<2cm，处理时间<5分钟/房间 |

### 2.2 数据产品化

| 产品模块 | 当前状态 | Phase 1目标 | 输出形态 |
|----------|----------|-------------|----------|
| 空间查询API | 5003端口，7端点 | 标准化RESTful API，支持Block/Road/Function/Object/Harness五层查询 | SDK v1.0 (Python/C++) |
| 约束检查API | 钻孔约束已跑通 | 扩展至拆改/搬运/消防/承重四大类约束 | API文档+测试用例 |
| 路径规划API | 基础版 | 支持跨楼层、室内外过渡的路径规划 | 路径JSON+可视化 |
| 材料查询API | 材料表已有 | 支持按物体ID/类别查询物理属性 | 标准化JSON输出 |
| 前端可视化 | 基础版 | 基于Spark 2.0/Three.js的3D场景展示 | Web Demo |

### 2.3 标杆Demo

选取黄葛渡或锦钰城一个完整地产项目，完成"从CAD图纸→四层模型→钻孔/搬运任务"的全链路闭环演示。

| Demo要素 | 具体内容 | 价值证明 |
|----------|----------|----------|
| 输入 | 真实CAD施工图（DWG） | 证明"我们能读懂施工图" |
| 处理 | 自动解析→四层模型构建→材料属性绑定→规范约束注入 | 证明"我们拥有工程级真值数据供应链" |
| 输出 | Agent执行钻孔/搬运任务，正确判断安全与否 | 证明"我们的数据能让机器人干对活" |
| 对比 | 有/无我们的API加持，任务成功率的差异 | 量化"Data-to-Action效率比" |
| 展现 | 3D可视化界面+录制Demo视频 | 融资/客户沟通材料 |

## 三、Phase 2：认知引擎（6→18月）

**核心命题**：打通多数据源融合，形成完整数据供应链，接入权威评测平台证明价值。

**报告对齐**："半结构化场景优先落地""多数据源长期共存""模型×数据供应链×部署闭环"

### 3.1 数据生成与增广

| 技术方向 | Phase 1基础 | Phase 2目标 | 产业对标 |
|----------|-------------|-------------|----------|
| 场景变体生成 | 100个合成场景 | 基于CAD场景自动生成1000+变体（光照/视角/杂物/季节） | 光轮智能的仿真数据策略 |
| 轨迹增广 | 手动遥操作数据 | 基于MimicGen/ROSIE自动扩增轨迹，覆盖更多操作变体 | 评测平台的专家轨迹集 |
| 物理仿真集成 | 未集成 | 对接Isaac Sim/MuJoCo，将CAD模型转化为可交互仿真场景 | 腾讯TAIROS的RoboFusion |
| FNO快速仿真 | 技术预研 | 完成2D热传导/楼板承重FNO验证，推理速度达传统方法100x | 实时物理反馈需求 |

## 四、Phase 3：生态引擎（18→36月）

**核心命题**：启动部署回流闭环，实现数据飞轮自运转，参与行业标准制定。

**报告对齐**："部署回流是最昂贵但也最有价值的下一轮数据源""缺少系统闭环能力，技术只能停留在演示Demo"

### 4.1 部署回流闭环

```
┌─────────────────────────────────────────────────────────────────┐
│               建成环境数据飞轮（Phase 3）                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│    ① 数据生产                              ④ 场景部署           │

│    + 真机数据采集                           ↓                   │

```

### 4.2 城市级扩展与标准制定

| 方向 | 具体内容 | 产业影响 |
|------|----------|----------|
| 覆盖范围扩展 | 从单一建筑→小区→园区→城市片区，整合规划图/道路CAD/市政管线/城市规范 | 成为智慧城市"建成环境数字底座" |
| 行业标准参与 | 联合腾讯/戴盟/评测平台，推动"建成环境具身数据标准"制定 | 定义行业规则 |
| 白皮书发布 | 《建成环境Physical AI数据规范与评测基准》 | 学术/产业权威背书 |
| 开源数据集 | 开源1000+场景建筑物理常识数据集 | 建立开发者生态 |

## 五、技术栈选型与演进路径

| 模块 | Phase 1 | Phase 2 | Phase 3 |
|------|---------|---------|---------|
| CAD解析 | ODA SDK + ezdxf | 自研CAD-AI引擎 | 云端解析服务（SaaS） |
| 几何计算 | Shapely + CGAL | PostGIS空间数据库 | 分布式空间计算集群 |
| 3D重建/渲染 | 3DGS (gsplat) | Spark 2.0 / Three.js | 云端流式渲染 |
| 后端API | FastAPI + SQLite | FastAPI + PostGIS + Redis | 微服务 + K8s |
| Agent框架 | QClaw (内测) | OpenClaw + TAIROS SDK | 多框架适配网关 |
| 仿真对接 | Isaac Sim基础 | Isaac Sim深度集成 + MuJoCo | 多仿真器兼容平台 |
| 物理加速 | - | FNO验证（2D热传导/承重） | 完整物理仿真加速器SaaS |
| AI/ML | PyTorch + Transformers | 自研空间编码器 + FNO | 垂直领域小模型 + 联邦学习 |
| 部署 | Docker单机 | Docker Compose集群 | K8s + 云原生 + 边缘节点 |
